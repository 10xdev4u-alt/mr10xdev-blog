From 10 users to 1000. The 4 stages, the bottlenecks, the
playbooks. Marketing, community, features, support. The growth
equation and when to focus on what.

## Stage 1: 0-10 users (week 1-4)

**Goal:** validate the agent does something useful.

**Activities:**
- Build the agent
- Use it yourself
- Show 5 friends
- Get 5 users

**Bottleneck:** the agent doesn't do anything useful yet.

**The fix:**
- Make the agent do 1 thing well
- Use it yourself for 1 week
- Show 1 person who has the same problem
- Iterate based on their feedback

**The metric:** are the 10 users using the agent daily? If not,
the agent doesn't solve a real problem.

## Stage 2: 10-100 users (month 2-6)

**Goal:** find product-market fit.

**Activities:**
- Open-source the agent
- Write a killer README
- Post on HN, Reddit, Twitter
- Respond to every issue and PR
- Add the obvious missing features

**Bottleneck:** discoverability. The agent is good but nobody
knows.

**The fix:**
- **Launch on Hacker News.** A "Show HN" post that explains
  the agent in 3 sentences. The technical audience on HN is
  your target market.
- **Launch on Reddit.** r/programming, r/MachineLearning,
  r/LocalLLaMA, etc. Each subreddit has different norms.
- **Tweet about it.** Twitter is the modern dev conference. Post
  about the agent, the design, the lessons learned.
- **Write a launch blog post.** Explain the agent, the use case,
  the alternatives. Link to the GitHub repo.
- **Ask your network.** 10 friends, 10 tweets, 10 messages.
  Each one becomes a user.

**The metric:** are the 100 users using the agent weekly? If
not, the agent doesn't have product-market fit yet.

## Stage 3: 100-1000 users (month 6-18)

**Goal:** build a community.

**Activities:**
- Write great documentation
- Add the 20% features (the edge cases)
- Add community channels (Discord, Discussions)
- Add a CoC and contributing guide
- Find co-maintainers
- Speak at conferences / write for publications

**Bottleneck:** community. The agent is good but the
contributions are slow, the issues pile up, the user gets
frustrated.

**The fix:**
- **Triage issues daily.** 30 minutes/day. Reply within 48
  hours. Close within a week.
- **Review PRs within 48 hours.** The contributor is excited
  for 48 hours. After that, the excitement fades.
- **Write a CoC and contributing guide.** Make it easy to
  contribute. Document the process.
- **Find 2-3 co-maintainers.** The project can't scale to one
  person. Recruit from the active contributors.
- **Host office hours.** 1 hour/week on Discord. Answer
  questions, get feedback, build relationships.
- **Speak at conferences.** The 5-minute lightning talk. The
  30-minute talk. Each one is a marketing event.
- **Write for publications.** Dev.to, freeCodeCamp, Hacker Noon.
  Each article is a marketing event.

**The metric:** are the 1000 users contributing back? Pull
requests, issues, discussions. The community is the moat.

## Stage 4: 1000+ users (year 2+)

**Goal:** sustain the project.

**Activities:**
- Add a governance model
- Add a foundation (or stay solo)
- Add a sustainability plan (sponsors, paid support)
- Add release management (LTS, deprecation policy)
- Add a security policy (CVE handling)

**Bottleneck:** sustainability. The project is successful but
the maintainer is burning out.

**The fix:**
- **Delegate.** Find 2-3 co-maintainers. Give them commit
  access. Trust them to make decisions.
- **Add a governance model.** BDFL, committee, or foundation.
  Document who decides what.
- **Find funding.** GitHub Sponsors, Open Collective, a
  foundation. The funding covers the maintainer's time.
- **Add a release cadence.** Weekly patch, monthly minor,
  quarterly major. The cadence is the contract.
- **Add a security policy.** SECURITY.md, private reporting,
  fast response. The security is the trust.

**The metric:** is the project growing without the maintainer?
The community is self-sustaining. The maintainer is a guide,
not a bottleneck.

## The growth equation

Users = (Launches × Reach × Conversion) × Retention

- **Launches:** how many places you tell the world
- **Reach:** how many people see each launch
- **Conversion:** what % try it
- **Retention:** what % keep using it

For 1000 users:
- 10 launches × 10K reach × 1% conversion × 10% retention = 100 users
- 100 launches × 10K reach × 1% conversion × 10% retention = 1000 users

The 100 launches is the hard part. The conversion and retention
come from the agent's quality.

## The 80/20

80% of the growth comes from:
- **Launches** (the initial marketing)
- **Word of mouth** (the existing users telling others)

20% comes from:
- **SEO** (the agent shows up in search)
- **Ads** (paid promotion)
- **Partnerships** (other projects link to yours)

For most indie projects, the 80% is enough. The paid growth
channels are for companies with budget.

## The anti-patterns

### Anti-pattern 1: Build it and they will come

The agent is good. The README is good. The docs are good.
Nobody uses it.

The fix: you have to tell the world. Launch on HN. Post on
Reddit. Tweet. Ask your network. Marketing is part of the
product.

### Anti-pattern 2: Spam

You post the agent on every subreddit, every Twitter thread,
every Discord. People notice the spam. They downvote you.

The fix: find the right channels. Be a good citizen. Add value
to the channels before promoting.

### Anti-pattern 3: Sell before you have product-market fit

You try to monetize before 100 users. The users leave. The
project dies.

The fix: 1000+ users first. Monetize later. The monetization
is the reward for building something people want.

## The lesson

4 stages. 1 equation. 1 lesson: launch, launch, launch.

The agent era is here. The growth is the same. The lessons
apply. Ship the agent. Launch it. Iterate. Grow. The agent
that survives is the one that's launched.

The growth is the work. The work is the discipline. The
discipline is what makes the project last.
