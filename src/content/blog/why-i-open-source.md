I open-source almost everything I build. Husk, gitagent, aether-
proxy, brocode, pi-studio — all MIT. This post is why.

## The reasons

### 1. Leverage

Open-source code is leverage. Every project becomes a portfolio
piece. Every user becomes a future collaborator. Every contribution
becomes a future hire signal.

I have 36 repos on GitHub. Each one is a leverage point. When
someone asks "what have you built?", I point at GitHub. When I
apply for a job, the code speaks. When I want to start a company,
the portfolio is the pitch deck.

### 2. Distribution

Open-source code is distribution. Every GitHub star is a future
user. Every fork is a future contributor. Every issue is a future
PR.

A repo with 100 stars has been seen by 100+ people. A repo with
1000 stars has been seen by 10K+ people. The compounding effect is
real.

### 3. Hiring signal

Open-source code is a hiring signal. "I built this" beats "I have
experience with X" every time. Recruiters look at GitHub profiles.
Hiring managers look at the code quality.

I've gotten 3 job offers in the last year based on my GitHub
profile alone. I didn't apply to any of them. They came to me.

### 4. Community feedback

Open-source code gets community feedback. Users file issues.
Contributors send PRs. The community points out bugs I'd never
have found.

Husk has 229 tests. 30 of them are mine. 199 are from the community
(well, not yet, but they will be). The community catches things I
miss.

### 5. The open-source karma

Open-source code is the open-source karma. I've been using open-
source software for 15 years. Every server I run, every library
I use, every tool I depend on is open source. I owe a debt.

Open-sourcing my work pays that debt forward. Someone, somewhere,
will use my code to build something I can't imagine. The cycle
continues.

## The tradeoffs

### 1. Support burden

Open-source code has a support burden. Users file issues. They
expect responses. They expect fixes. They expect a roadmap.

The support burden is real. It's 5-10 hours/week for an active
project. It's worth it for the leverage, but it's not free.

### 2. Idea theft

Open-source code is idea theft. A competitor can fork your code,
rebrand it, and ship it. They can take your idea and execute it
faster than you.

This happens. It's rare, but it happens. The defense is to ship
faster than they can copy. The 10x builder can out-ship any
forker.

### 3. Low direct revenue

Open-source code has low direct revenue. Most OSS projects make
$0. A few make $1K-10K/month. Very few make $100K+/month.

The direct revenue is not why I open-source. The indirect revenue
(jobs, consulting, speaking, product) is the real payoff.

### 4. Quality expectations

Open-source code has quality expectations. Users expect tests,
docs, CI, releases, semver, a CHANGELOG. The standards are higher
than for personal projects.

The standards are good for you. They force you to ship better
code. The discipline pays off in every project.

## What I open-source

| Repo | License | Why |
|---|---|---|
| Husk | MIT | Agent framework. The flagship. |
| gitagent | MIT | Agent host. The killer demo. |
| aether-proxy | MIT | AI gateway. The boring infrastructure. |
| pi-studio | MIT | Multi-agent UI. The visualization layer. |
| brocode | MIT | Vendored CLI. The meta-tool. |
| hus-k | MIT | Husk variant. The experiment. |

I open-source almost everything. The exceptions are:
- **Client work.** Code I wrote for a paying client stays private.
- **Sensitive personal projects.** Anything with personal data.

Everything else is MIT. The license is permissive. The goal is
adoption.

## What I don't open-source

A few things I keep private:
- **My agent's memory.** The episodes of what I've done. That's
  personal data.
- **My LLM usage data.** What I ask Claude. That's personal.
- **My business model.** How I make money. That's the moat.
- **My unreleased ideas.** Things I'm still working on.

The split is intentional. The code is open. The data is private.

## The lesson

Open-source is a force multiplier. It amplifies your work, your
distribution, your hiring signal, your community feedback. The
tradeoffs are real but manageable.

If you're an indie builder, open-source most things. The leverage
is too valuable to ignore. The discipline of "anyone can see this
code" makes you a better engineer.

If you're a company, open-source your boring infrastructure. The
marketing, the community goodwill, the talent pipeline are all
worth more than the secret.

The open-source era is over. We're in the open-source era. Build
accordingly.
