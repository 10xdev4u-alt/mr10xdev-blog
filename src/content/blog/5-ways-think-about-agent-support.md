5 ways to think about support: docs, community, paid,
dedicated, none. Each is a different support model. The
framework, the examples, the lesson.

## Model 1: Docs

The support is via docs. The docs are the truth. The docs
are the value.

**Examples:**
- A README with quick start
- A reference with API
- A guide with how-to

**When to use:** the user is self-sufficient. The user
is technical. The user is in the long tail.

**Pros:**
- Scalable (the docs scale to infinity)
- Cheap (the docs cost time once)
- Standard (the docs are universal)

**Cons:**
- Effort (the docs take effort to write)
- Limited (the docs can't answer every question)
- Stale (the docs can be outdated)

## Model 2: Community

The support is via community. The community is the truth.
The community is the value.

**Examples:**
- A Discord channel
- A GitHub Discussions thread
- A subreddit

**When to use:** the user is engaged. The user is in the
community. The user is a fan.

**Pros:**
- Engaged (the community is engaged)
- Word-of-mouth (the community spreads the word)
- Cheap (the community is free)

**Cons:**
- Effort (the community takes effort to maintain)
- Slow (the community is slow to respond)
- Risk (the community can be toxic)

## Model 3: Paid

The support is paid. The user pays for support. The user
is the truth.

**Examples:**
- "Email support: $99/month"
- "Priority support: $299/month"
- "24/7 support: $999/month"

**When to use:** the user is business. The user wants
quality. The user is committed.

**Pros:**
- Quality (the support is quality)
- Revenue (the support is revenue)
- Aligned (the support is aligned with the user)

**Cons:**
- Cost (the support costs the user)
- Friction (the support has friction)
- Limit (the support is limited to paying users)

## Model 4: Dedicated

The support is dedicated. The user has a dedicated
support engineer. The user is the truth.

**Examples:**
- "Your dedicated CSM"
- "Your dedicated engineer"
- "Your dedicated team"

**When to use:** the user is enterprise. The user wants
custom. The user is large.

**Pros:**
- Custom (the support is custom)
- Quality (the support is high quality)
- Long-term (the support is long-term)

**Cons:**
- Cost (the support is the most expensive)
- Limit (the support is limited to large customers)
- Risk (the support depends on the engineer)

## Model 5: None

The support is none. The user is on their own. The user
is the truth.

**Examples:**
- "No support"
- "Best effort"
- "Use at your own risk"

**When to use:** the user is small. The user is on their
own. The agent is a hobby.

**Pros:**
- Cheap (the support is free)
- Simple (the support is simple)
- Fast (the support is fast to set up)

**Cons:**
- Risky (the user is on their own)
- Friction (the user has friction)
- Limit (the user can't get help)

## The 5 together

The 5 are the models. The models are the support. The
support is the value.

| Model | Quality | Cost | Best for |
|---|---|---|---|
| Docs | Medium | Low | Long tail |
| Community | Medium | Low | Engaged |
| Paid | High | High | Business |
| Dedicated | Highest | Highest | Enterprise |
| None | Low | Free | Hobby |

The model that matches the user is the right model.

## The 80/20

80% of the value comes from:
- Docs (the user is self-sufficient)
- Community (the user is engaged)

20% comes from:
- Paid (the user is business)
- Dedicated (the user is enterprise)
- None (the user is on their own)

Focus on the 80% first. Add the 20% as you grow.

## The choice

For each agent, ask:
- Is the user self-sufficient? (docs)
- Is the user engaged? (community)
- Is the user business? (paid)
- Is the user enterprise? (dedicated)
- Is the user on their own? (none)

The right answer is the right model at the right cost.

## The lesson

5 models. 1 support. 1 lesson: pick the right one.

The model that matches the user is the right model. The
model that doesn't match is the wrong model.

The agent era is here. The support is the design. The
design is the choice. The choice is the discipline.
