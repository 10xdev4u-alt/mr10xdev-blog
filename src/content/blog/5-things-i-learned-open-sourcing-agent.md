5 things I learned from open-sourcing my agent. The
surprises, the lessons, the truth. The hard truths from
the trenches.

## Lesson 1: People will read the README first

The README is the front door. The README is the first
impression. The README is the truth.

**What I expected:** "People will read the docs. People
will dig deep."

**What it was:** "People will read the README for 30
seconds. If the README is good, they'll try. If the README
is bad, they'll leave."

The README is the truth. The README is the first
impression. The README is the conversion.

**The lesson:** invest in the README. The README is the
product.

## Lesson 2: People will file issues, not PRs

The user will file an issue. The user will not file a PR.
The user will wait for someone to fix it.

**What I expected:** "People will contribute. People will
file PRs."

**What it was:** "People will file issues. People will
wait. People will be frustrated."

The issue-to-PR ratio is 10:1 or worse. The user is a
user, not a contributor.

**The lesson:** design for the user. The user files
issues. The maintainer files PRs.

## Lesson 3: People will use the agent in ways you didn't expect

The user will use the agent in a way you didn't
anticipate. The user will use the agent for a use case
you didn't think of. The user will surprise you.

**What I expected:** "People will use the agent for the
use cases I designed."

**What it was:** "People will use the agent for use cases
I never imagined. Some use cases will be brilliant. Some
will be bizarre."

The user is creative. The user will surprise you.

**The lesson:** design for the unexpected. The user will
surprise you.

## Lesson 4: People will complain about the wrong things

The user will complain about the thing you don't think
matters. The user will be silent about the thing you
spent weeks on.

**What I expected:** "The user will love the features I
worked hard on."

**What it was:** "The user will love the simplest thing.
The user will hate the most complex thing."

The user's priorities are not the maintainer's priorities.
The user is the truth.

**The lesson:** listen to the user. The user knows what
matters.

## Lesson 5: People will thank you

The user will thank you. The user will appreciate the
work. The user will send you a message saying "this
saved me hours."

**What I expected:** "The user will be silent. The user
won't say thanks."

**What it was:** "The user will thank me. The user will
be grateful. The user will make my day."

The user is grateful. The user is the reason you build.

**The lesson:** remember the thanks. The thanks is the
fuel.

## The 5 lessons together

The 5 are the truth. The truth is the discipline. The
discipline is the open source.

| Lesson | What it taught me |
|---|---|
| README is the front door | Invest in the README |
| Issues, not PRs | Design for the user |
| Unexpected uses | Design for the unexpected |
| Wrong complaints | Listen to the user |
| Thanks | Remember the thanks |

The lessons are the hard truths. The hard truths are the
lessons. The lessons are the discipline.

## The 80/20

80% of the value comes from:
- README is the front door (the user decides in 30 seconds)
- Thanks (the user is the fuel)

20% comes from:
- Issues, not PRs (the user is a user)
- Unexpected uses (the user is creative)
- Wrong complaints (the user is the truth)

Focus on the 80% first. Add the 20% as you grow.

## The lesson

5 lessons. 1 open source. 1 lesson: the user is the truth.

The maintainer that listens to the user is the maintainer
that succeeds. The maintainer that doesn't is the
maintainer that fails.

The agent era is here. The user is the design. The design
is the discipline. The discipline is the success.
