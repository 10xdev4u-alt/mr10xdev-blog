I started building Husk on a Tuesday. By Friday it had 9 releases and
229 tests. Here's why.

## The problem

Every AI agent framework I tried in 2025 was either:

1. **A thin wrapper around an LLM.** "Call Claude and stream the
   response." Useful, but not a framework.
2. **A heavy orchestration layer.** LangChain, LlamaIndex, AutoGen.
   Useful, but 200+ dependencies and a steep learning curve.
3. **A "no-code" agent builder.** CrewAI, Voiceflow. Useful for
   non-developers, but you can't extend them.

I wanted option 4: **a small, typed, unopinionated TypeScript library
that does one thing well — give me an `Agent` I can call.**

## The principles

Three principles guided the design:

### 1. Small surface

If a feature can live outside the core, it lives outside the core.
The core is:
- A `chat()` and `stream()` function
- A `ToolDefinition` interface
- A `VectorStore` interface
- A `Memory` interface
- An `Agent` class that ties them together

Everything else is an adapter, an example, or a plugin. The core
stays under 50KB.

### 2. Types are the docs

Every public API is a TypeScript type. No `any`. No implicit. The
`.d.ts` is the documentation; the implementation is a side effect.

If I can't express the API in TypeScript, I don't understand it well
enough to ship it.

### 3. The 80/20 of agent design

The 80% of agent use cases are:
- Read input (issue body, PR diff, comment)
- Call LLM
- Run a tool
- Post output (comment, label, PR)

The 20% is everything else. v0.1.0 ships the 80%. The 20% is opt-in
via plugins.

## The anti-patterns

I explicitly did NOT include:
- A "chain" abstraction (LangChain's bread and butter — but chains
  are usually a YAGNI trap)
- A "memory" framework (different problems need different memory; a
  small interface + 2-3 implementations is enough)
- A "vector DB" abstraction (each one has a unique API; abstractions
  leak)
- A "prompt template" DSL (just use template literals)
- A "deployment" story (deploy it like any other Node app)

## The result

Nine releases, 229 tests, ~60KB lib bundle. Three providers
(Anthropic, OpenAI, Ollama, with a fourth — Gemini — coming in
v0.8.0). 12 examples. A real-world agent (this blog) running on it.

The point was never to be the biggest framework. The point was to be
the one I'd want to use.

## The 80/20 of agent design (revisited)

After shipping, I realize the real 80% is:
- A clear input/output contract
- A typed tool framework
- A retry loop with backoff
- An approval gate for write actions
- A cost tracker

Everything else is decoration. Husk v0.5.0+ is just adding the
remaining 5%.

## What's next

The natural next step is a meta-loop: an agent that maintains
Husk. The CHANGELOG is generated by an agent. The doc updates are
generated by an agent. The release is triggered by an agent.

That's what [gitagent](https://github.com/10xdev4u-alt/gitagent) is
for. And this very site is the demo.
